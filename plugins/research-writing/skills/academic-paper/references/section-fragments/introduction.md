# Section: Introduction

## Default Funnel

`field scale -> bottleneck -> prior attempts -> unresolved gap -> present study`

## Paragraph Jobs (typical 4-paragraph intro)

1. Establish the field stake. Make it land for a non-specialist if the target is broad.
2. Identify the bottleneck in existing practice.
3. Summarize what prior work has and has not solved. Synthesize, do not list.
4. State what this paper does and how it addresses the gap. Preview the contribution, not the results in detail.

## Nature Summary Paragraph Pattern

For broad-audience venues (Nature, Science), strengthen the funnel into:

`broad field -> sharper background -> exact problem -> here we show -> what the result changes -> broader context / outlook`

### Sentence-level structure

1. **Broad field introduction** (1-2 sentences) — orient any scientist; establish why the field matters; avoid dense terminology early.
2. **More detailed background** (2-3 sentences) — move from field-scale context to the specific system/process; introduce minimum specialist concepts needed to understand the study.
3. **General problem statement** (1 sentence) — name the exact unresolved problem; create the hinge between background and contribution. Form: `However, ... remains unknown` or `Yet whether ... is unclear`.
4. **Main result sentence** (1 sentence) — state the central finding clearly using `Here we show` or equivalent. Express the main claim, not the whole method pipeline.
5. **Direct implications** (2-3 sentences) — explain what the main result reveals; compare with what was previously believed; show how the result adds to prior knowledge. This is the intellectual center.
6. **General context** (1-2 sentences) — place the result back into a broader methodological or translational context; anchored in what the study actually supports.

## Pipeline Variants

For method-heavy papers, the introduction often follows a different pattern:

- **task-then-application** — define a task abstractly, then show its applications
- **application-first** — open with a concrete application or pain point, narrow to the task
- **general-to-specific setting** — broad field -> specific subproblem
- **open-with-challenge** — lead with the unsolved difficulty
- **novel-task-challenge-decomposition** — define a new task, decompose its challenges
- **technical-challenge** (3 variants) — frame around the technical bottleneck, then the move
- **pipeline-version** (4 variants) — for module/pipeline papers, frame contribution as one or several modules

Tell the reader which variant you picked and why.

## Technical Challenge Framing

### Version 1: Existing task with existing methods

Start with a general challenge -> traditional methods and limitations -> recent methods and limitations with technical reasons -> the final limitation is exactly the challenge your method solves.

### Version 2: Existing task with insight rooted in traditional methods

Start from mainstream methods -> classical/traditional line containing similar insight -> why it's still insufficient -> modern methods and unresolved technical reason -> bridge to your method.

### Version 3: Novel task without direct prior methods

State the goal -> explain the problem is challenging for N reasons -> separate challenge points (First/Second/Finally) -> transition to pipeline.

## Contribution Presentation Patterns

### Pipeline Version 1: One contribution, multiple advantages
Introduce core framework -> point to teaser figure -> key novelty sentence -> specific implementation -> multiple advantages.

### Pipeline Version 2: Two contributions
Framework + key novelty -> teaser figure -> contribution 1 + advantage -> remaining challenge -> contribution 2.

### Pipeline Version 3: New module on existing pipeline
Prior pipeline setup -> new module as key innovation -> observation motivating the module -> module mechanism -> comparison with alternatives.

### Pipeline Version 4: Observation-driven
Key innovation first -> intuitive observation as motivation -> implementation details -> technical advantage and empirical gain.

## Drafting Rules

- Do not summarize results in detail. The final paragraph states the contribution and approach, not the numbers.
- Cite prior work to position, not to demonstrate breadth. Each citation should earn its place.
- The transition from "what is known" to "what this paper does" must be explicit, not implied.
- Do not first present a naive solution and then describe your improvement over it. That makes the work look incremental even if it isn't.

## Common Failure Modes

- Starting too narrow — gene/protein/material/model names before any field context.
- No clear gap sentence — background slides directly into contribution.
- `Here we show` comes too early (reader lacks setup) or too late (reads like a review).
- Result listing without interpretation — findings are listed but never explained collectively.
- Overextended final significance — jumps to societal/clinical importance unsupported by evidence.

## Diagnostics Before Finalizing

- Does the first sentence of each paragraph state its message?
- Are technical challenge, technical reason, and solved mechanism all explicit?
- Are claims in Introduction aligned with experiment evidence?
- Is terminology stable across all sections?
